# LLMに手伝ってもらって
# Blender保存時glTF書き出しでライブプレビュー

hello-blender-three-live-preview

---

## なぜこの話をするのか

Blender と Three.js をつないで
確認を速くしたかったからです。

やりたかったのは、

- Blender で編集する
- 書き出す
- ブラウザで確認する

この往復をできるだけ軽くすることでした。

---

## 最初に考えた形

狙いはシンプルでした。

- Blender を保存したら
- glTF を自動で書き出して
- Three.js 側でライブプレビューする

書き出しの手間がなくなれば、
試行回数も増やせるはずでした。

---

## そこで LLM を使った

今回は本筋ではない部分の効率化を、
LLM に手伝ってもらいました。

特にこの部分は周辺作業なので、
API を追うところから
自分で抱え込みたくなかったです。

そこで、
調査や叩き台づくりを
LLM に手伝ってもらいました。

- Blender Python のたたき台
- export 周りの調査
- 実装の叩き方の相談

着手のハードルを下げる助けになりました。

クラッシュしたときにも、
手元のコードだけでなく
公開されている Blender の実装知識も踏まえて
ログを読み解き、
修正方針まで素早くたどり着けたのは印象的でした。

---

## でも保存時実行には問題があった

保存時に自動で処理を走らせると、
エラーが出ていても気づきにくいです。

- 保存の裏で動く
- 失敗が見えにくい
- そのままクラッシュにつながる

便利にしたかった仕組みが、
逆に扱いづらくなりました。

---

## つらかったのは原因が追いにくいこと

単に失敗するだけではなく、
何が悪いのか分かりにくいのが問題でした。

- 保存操作に埋もれる
- 失敗地点が見えにくい
- 再現と切り分けがしづらい

自動化のつもりが、
デバッグ性を下げていました。

---

## このリポジトリで見ていた場所

- `blender/`
  Blender 関連のファイル
- `blender/scripts/auto_export_gltf.py`
  export 処理の中心
- `src/`
  Three.js 側の実装
- `public/models/`
  書き出されたモデル

接続自体は単純でも、
実行タイミングの設計が重要でした。

---

## 落とし所はボタン実行だった

最終的には、
保存時自動実行より
ボタンで明示的に実行する方がよいと考えています。

- いつ動くか分かる
- 失敗に気づける
- 原因を追いやすい

少し手間は増えても、
かなり安全です。

---

## 今回の学び

- 自動化は強い
- でも失敗が見えにくい場所に置くと危険
- 本筋ではない周辺作業は LLM に手伝ってもらうと進めやすかった
- 便利さだけで自動化しない

大事なのは、
どこを自動化するかでした。

---

## まとめ

- 保存時自動 export は魅力的だった
- ただし観測しづらく、クラッシュも起きた
- 今回はボタン実行が現実的な落とし所だった
- LLM には API 調査や叩き台づくりを手伝ってもらった
