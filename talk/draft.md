# タイトル

LLMに手伝ってもらってBlender保存時glTF書き出しでライブプレビュー

## ひとことで言うと

LLM を補助役として使いながら、Blender 保存時の glTF 自動書き出しを試したものの、エラーに気づきにくくクラッシュも起きたので、最終的にボタン実行が落とし所だと判断した話。

## 想定時間

5 分

## 草稿

こんにちは。今日は、このリポジトリを題材に、LLM に手伝ってもらいながら Blender と Three.js の往復を軽くしようとして、最終的に「保存時自動実行ではなくボタン実行にした」という話をします。

最初にやりたかったのは、Blender 側でモデルを編集して保存したタイミングで glTF もしくは glb を書き出し、Three.js 側でその結果をすぐ確認できるようにすることでした。

この手の作業では、モデリングツールと実行環境の間を何度も行き来するので、書き出しが面倒だとすぐ試行回数が落ちます。このリポジトリでは、その面倒さを減らす方向に寄せています。

そのときに効いたのが LLM です。ゼロから全部を書かせるというより、自動 export のたたき台や Blender Python の細部を一緒に詰める補助役として使うことで、試しながら前に進めやすくなりました。

ただ、保存時に実行する方式には問題がありました。エラーが出ていても保存操作の裏で走るので気づきにくく、しかもその状態がクラッシュにつながっていました。何が原因なのかも追いにくく、便利にしたかったはずの仕組みが逆にデバッグしづらくなっていました。

リポジトリの構成は比較的わかりやすくて、Blender 関連のファイルは `blender/`、Web 側の実装は `src/`、書き出されたモデルは `public/models/` にあります。

特にポイントになるのは、`blender/scripts/auto_export_gltf.py` のような書き出しスクリプトと、Three.js 側の読み込み処理です。接続自体はシンプルですが、実行タイミングの設計がかなり重要でした。

保存時自動実行は一見便利ですが、失敗が見えにくい処理を裏で走らせることになるので、少なくとも今回のような個人開発の小さな仕組みでは扱いが難しかったです。

そこで、最終的にはボタンで明示的に export を実行する形が落とし所だと考えています。これなら、いつ処理が走るかが明確で、失敗しても気づきやすく、原因の切り分けもしやすいです。

加えて、LLM を使うことで Blender API や export 周りの調査コストを少し下げられました。実装の責任は自分で持ちつつ、調べ物と叩き台づくりを早める使い方が相性よかったです。

今回の学びは、自動化は強いけれど、失敗が観測しにくくなる場所に入れると一気に扱いづらくなるということです。便利さだけで保存時実行に寄せるより、明示的なボタン実行の方が、結果として安全で運用しやすいと感じました。

ありがとうございました。
