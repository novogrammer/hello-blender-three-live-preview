# タイトル

LLMに手伝ってもらってBlender保存時glTF書き出しでライブプレビュー

## ひとことで言うと

LLM を補助役として使いながら、Blender 保存時に glTF を書き出して Three.js 側でライブプレビューできる開発ループを作った話。

## 想定時間

5 分

## 草稿

こんにちは。今日は、このリポジトリを題材に、LLM に手伝ってもらいながら Blender と Three.js の往復をできるだけ軽くするための小さな仕組みを紹介します。

やっていることはシンプルで、Blender 側でモデルを編集して保存したタイミングで glTF もしくは glb を書き出し、Three.js 側でその結果をすぐ確認できるようにしています。

この手の作業では、モデリングツールと実行環境の間を何度も行き来するので、書き出しが面倒だとすぐ試行回数が落ちます。このリポジトリでは、その面倒さを減らす方向に寄せています。

そのときに効いたのが LLM です。ゼロから全部を書かせるというより、自動 export のたたき台や Blender Python の細部を一緒に詰める補助役として使うことで、試しながら前に進めやすくなりました。

リポジトリの構成は比較的わかりやすくて、Blender 関連のファイルは `blender/`、Web 側の実装は `src/`、書き出されたモデルは `public/models/` にあります。

特にポイントになるのは、`blender/scripts/auto_export_gltf.py` のような保存時の自動書き出しスクリプトと、Three.js 側の読み込み処理です。この 2 つがつながることで、アセット確認のループが短くなります。

この仕組みの良いところは、巨大なパイプラインを作らなくても、個人開発や検証用途で十分に効果があることです。まずは小さくつないで、必要なら後で監視や最適化を足せます。

加えて、LLM を使うことで Blender API や export 周りの調査コストを少し下げられました。実装の責任は自分で持ちつつ、調べ物と叩き台づくりを早める使い方が相性よかったです。

もし Blender と WebGL の往復が面倒だと感じているなら、全部を自動化しなくても、最初の一歩として「保存時の書き出しを自動化する」だけでかなり体験が変わると思います。さらに、その実装を進める補助として LLM を使うと、着手のハードルも下げやすいです。

ありがとうございました。
